## About

My personal portfolio, built to showcase my work in system administration, infrastructure automation, and security. Built from scratch with vanilla HTML, CSS, and JavaScript — no frameworks, no build step required to run.

The UI is styled as a desktop OS environment with draggable modal windows, a dark/light theme toggle, and a built-in markdown blog reader for technical notes.

**Live site:** [sminaya-dev.github.io](https://sminaya-dev.github.io)

---

## Features

- **OS-style window system** — modal windows are draggable on desktop, with z-index stacking and position memory across toggles
- **Responsive layout** — adapts to mobile and tablet viewports; inline styles are stripped and windows are reset on breakpoint changes
- **Dark/light theme** — respects `prefers-color-scheme` on first load, persists selection to `localStorage`, and re-highlights code blocks on switch
- **Markdown blog reader** — fetches and renders `.md` posts from a `manifest.json` index, with syntax highlighting via highlight.js and prev/next navigation
- **Local image resolution** — custom marked.js renderer automatically resolves relative image paths inside post content

## Technical Highlights

**Module-based vanilla JS**

The site uses native ES modules (`type="module"`) with no bundler. Each concern is isolated — `state.js` holds shared state, `window-manager.js` owns all open/close/positioning logic, `blog.js` handles all post fetching and rendering, and `theme.js` manages the theme lifecycle independently. `main.js` wires them together at startup.

**Draggable window system**

Windows spawn at calculated offsets relative to the main container rather than fixed viewport coordinates, so positioning adapts naturally to different screen sizes. Drag position is saved to `state.windowPositions` so toggling a window closed and back open restores its last location. A bounds check prevents windows from being dragged off-screen.

**Responsive breakpoint handling**

On resize, the screen size is recalculated and stored in `state.screenSize`. When dropping below desktop width, `resetStylesForMobile()` strips all inline `left`/`top`/`transform` styles from modals so CSS media queries can take over — avoiding a conflict between JS-applied positioning and responsive CSS.

**Markdown rendering pipeline**

Posts are plain `.md` files with YAML front matter (`title`, `date`). `build-posts.js` scans the `posts/` directory locally and writes a `manifest.json` index — the site fetches this at runtime to populate the notes list without any server-side logic. Front matter is stripped before passing content to marked.js, and a custom image renderer resolves relative paths to the correct `posts/` prefix automatically.

**Theme-aware syntax highlighting**

highlight.js stylesheet is swapped at runtime when the theme changes. Already-rendered code blocks are re-highlighted by clearing the `data-highlighted` attribute and calling `hljs.highlightElement()` again — ensuring code blocks stay visually consistent after a theme switch.

## Blog Posts

Posts live in the `posts/` directory as `.md` files with front matter:

```markdown
---
title: Your Post Title
date: 2026-01-15
---

Post content here...
```

To add a new post, drop the `.md` file into `posts/` and run:

```
node build-posts.js
```

This regenerates `manifest.json`. Commit both the new post and the updated manifest.
